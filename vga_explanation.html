1. VGA 동작 원리

VGA 신호란?

VGA(Video Graphics Array)는 아날로그 RGB 신호와 두 개의 동기 신호(Hsync, Vsync)로 화면을 구성합니다. 모니터는 이 신호를 받아 전자빔(또는 픽셀 행렬)으로 화면을 좌→우, 위→아래 순서로 그립니다.

래스터 스캔(Raster Scan) 방식

VGA는 래스터 스캔 방식으로 동작합니다. 전체 화면은 한 픽셀씩 좌에서 우로 스캔하고, 한 줄이 끝나면 다음 줄로 내려가는 것을 반복합니다.

수평 스캔 (1줄 = 800 픽셀 클럭)

  • VISIBLE 640 픽셀 — 실제 화상 출력
  • FRONT 16 픽셀 — 앞 포치(blank)
  • HSYNC 96 픽셀 — 동기 펄스(active-low)
  • BACK 48 픽셀 — 뒤 포치(blank)

수직 스캔 (1프레임 = 525 라인)

  • VISIBLE 480 라인 — 실제 화상 출력
  • FRONT 10 라인 — 앞 포치
  • VSYNC 2 라인 — 수직 동기 펄스(active-low)
  • BACK 33 라인 — 뒤 포치

픽셀 클럭 계산

표준 VGA 640×480 @ 60Hz 기준 픽셀 클럭: 25.175 MHz

본 프로젝트는 100 MHz 시스템 클럭을 사용하므로, 내부적으로 pix_phase 카운터(0~3)로 4분주하여 픽셀당 4클럭 = 실효 25 MHz를 구현합니다.

픽셀 클럭 = 100MHz / 4 = 25MHz
프레임 레이트 = 25,000,000 / (800 × 525) ≈ 59.5 Hz

출력 신호 구성

신호비트 수설명
vgaRed4비트적색 채널 (RGB444 포맷)
vgaGreen4비트녹색 채널
vgaBlue4비트청색 채널
Hsync1비트수평 동기 (active-low)
Vsync1비트수직 동기 (active-low)
Active-low 동기 신호: Hsync와 Vsync는 평상시 High(1)를 유지하다가 동기 펄스 구간에서만 Low(0)로 떨어집니다. 모니터는 이 Low 구간을 기준으로 줄 시작/화면 시작을 인식합니다.

두 가지 렌더링 모드

문자 모드 (canvas_mode = 0)

화면을 40열 × 30행 = 1200개 문자 셀로 나눕니다. 각 셀은 16×16 픽셀이며, font_rom에서 해당 ASCII 코드의 글리프 비트맵을 읽어 전경색/배경색으로 렌더링합니다.

캔버스 프리뷰 모드 (canvas_mode = 1)

화면 좌상단 (x=16, y=48) 기준 224×224 픽셀 영역에 28×28 손글씨 캔버스를 8배 확대(픽셀당 8×8 블록)하여 표시합니다. 나머지 영역은 문자 모드로 계속 렌더링됩니다.

3. vga_ctrl.v — VGA 렌더링 엔진

VGA 실제 동작의 핵심 모듈입니다. 픽셀 위치 카운터, 문자/캔버스 버퍼, 폰트 ROM 조회, 색상 렌더링, 동기 신호 생성을 모두 담당합니다.

모듈 파라미터 및 포트

module vga_ctrl #(
    parameter FONT_MEM_FILE = "font_rom.mem"  // 폰트 비트맵 파일 경로
)(
    // 시스템 클럭 / 리셋
    input  clk, reset_p,

    // 제어 입력
    input  enable,       // 0=화면 꺼짐(전체 검정), 1=정상 출력
    input  clear,        // 1-클럭 펄스: 문자버퍼 전체를 space(0x20)로 초기화
    input  canvas_mode,  // 0=문자 전용, 1=캔버스 프리뷰 합성

    // 문자 버퍼 Write 포트
    input [10:0] char_wr_addr,  // 0~1199 (40열 × 30행)
    input  [7:0] char_wr_data,  // ASCII 코드
    input        char_wr_en,

    // 캔버스 버퍼 Write 포트
    input  [9:0] canvas_wr_addr, // 0~783 (28×28)
    input        canvas_wr_data, // 1=흰, 0=검
    input        canvas_wr_en,

    // 색상
    input [11:0] fg_color,       // RGB444 전경색
    input [11:0] bg_color,       // RGB444 배경색

    // VGA 출력
    output reg [3:0] vgaRed, vgaGreen, vgaBlue,
    output reg       Hsync, Vsync,
    output wire      clear_busy   // clear 진행 중 플래그
);

내부 버퍼 구조

문자 버퍼 (char_buf)

reg [7:0] char_buf [0:1199]

1200개 ASCII 코드를 저장. distributed RAM으로 추론됩니다 (포트 B 읽기 1클럭 지연).

  • 주소 = char_row × 40 + char_col
  • row = v_cnt[8:4] (0~29)
  • col = h_cnt[9:4] (0~39)

캔버스 버퍼 (canvas_buf)

reg canvas_buf [0:783]

784개 1비트 픽셀. 28×28 손글씨 영역을 저장합니다.

  • 주소 = preview_y × 28 + preview_x
  • 프리뷰 픽셀 = 8×8 블록으로 확대 표시

3단계 파이프라인 (pix_phase 0→1→2→3)

폰트 ROM이 블록 RAM(1클럭 레이턴시)이므로, 픽셀 데이터를 미리 준비하기 위해 4-phase 파이프라인을 사용합니다.

Phase동작신호
0char_buf / canvas_buf 읽기 요청 (combinatorial)char_code_q, canvas_pixel_q 래치
1폰트 ROM 주소 계산 및 요청 전달visible_d, char_x_d, canvas_pixel_d 래치
2폰트 ROM 출력 대기 (1클럭 레이턴시)font_data 안정화
3최종 색상 결정 + h/v 카운터 증가 + 동기 신호 출력vgaRed/Green/Blue, Hsync, Vsync

렌더링 우선순위 (Phase 3)

if (!enable || !visible_d) begin
    render_color = 12'h000;           // 꺼짐 또는 blanking 구간 → 검정
end else if (in_preview_d) begin
    render_color = canvas_pixel_d ? 12'hFFF : 12'h000;  // 캔버스: 흰/검
end else if (text_pixel_on) begin
    render_color = fg_color;           // 폰트 비트=1 → 전경색
end else begin
    render_color = bg_color;           // 폰트 비트=0 → 배경색
end

clear 동작

1클럭 clear 펄스가 들어오면 clear_active 플래그가 Set되고, clear_idx가 0에서 1199까지 증가하면서 매 클럭 char_buf[clear_idx] ← 0x20(space)를 씁니다. 1200 클럭(12μs @ 100MHz) 후 완료되면 clear_active가 해제됩니다.

캔버스 프리뷰 주소 계산

// 프리뷰 영역: 화면상 (16,48) ~ (240,272), 224x224 픽셀
wire [4:0] preview_x = (h_cnt - PREVIEW_X0) >> 3;  // 8픽셀 = 1셀
wire [4:0] preview_y = (v_cnt - PREVIEW_Y0) >> 3;
wire [9:0] canvas_rd_addr =
    ({5'd0, preview_y} << 5) - ({5'd0, preview_y} << 2) + {5'd0, preview_x};
// = preview_y * 28 + preview_x  (32 - 4 = 28 계산)

동기 신호 생성

// Hsync: h_cnt가 656~751 구간에서 Low
Hsync <= ~((h_cnt >= (H_VISIBLE + H_FRONT)) &&
            (h_cnt <  (H_VISIBLE + H_FRONT + H_SYNC)));
// Vsync: v_cnt가 490~491 구간에서 Low
Vsync <= ~((v_cnt >= (V_VISIBLE + V_FRONT)) &&
            (v_cnt <  (V_VISIBLE + V_FRONT + V_SYNC)));

4. vga_axi.v — AXI4-Lite 슬레이브 래퍼

MicroBlaze 프로세서(또는 PS)가 AXI4-Lite 버스로 VGA 컨트롤러를 제어할 수 있도록 레지스터 맵을 제공하는 래퍼 모듈입니다. vga_ctrl을 내부에 인스턴스화합니다.

레지스터 맵

오프셋이름비트 필드설명
0x00CTRL[0]=enable, [1]=clearVGA 활성화, 화면 지우기 트리거
0x04CHAR_ADDR[10:0]쓸 문자 버퍼 위치 (0~1199)
0x08CHAR_DATA[7:0]ASCII 문자 코드
0x0CWR_STRB[0]=trigger1을 쓰면 char_wr_en 펄스 생성
0x10FG_COLOR[11:0]전경색 RGB444
0x14BG_COLOR[11:0]배경색 RGB444
0x18CANVAS_WR_ADDR[9:0]캔버스 쓰기 주소 (0~783)
0x1CCANVAS_WR_DATA[0]캔버스 픽셀값 (0 또는 1)
0x20CANVAS_WR_EN[0]=trigger1을 쓰면 canvas_wr_en 펄스 생성
0x24CANVAS_MODE[0]0=문자모드, 1=캔버스 합성모드

AXI4-Lite Write 핸드셰이크

// AWVALID && WVALID && !axi_awready (이전 응답이 없는 상태)일 때만 처리
wire write_fire = S_AXI_AWVALID && S_AXI_WVALID &&
                   !axi_awready && !axi_wready && !axi_bvalid;

if (write_fire) begin
    axi_awready <= 1'b1;  // 주소 수락
    axi_wready  <= 1'b1;  // 데이터 수락
    axi_bvalid  <= 1'b1;  // 쓰기 응답 준비
    // case로 레지스터 주소 디코딩 및 값 저장
end

펄스 신호 생성 방식

char_wr_en, canvas_wr_en, clear는 1클럭 펄스로만 동작해야 합니다. AXI 쓰기 사이클마다 매 클럭 초기화하고, 해당 레지스터 주소에 1이 쓰일 때만 1로 Set합니다:

// 매 클럭 초기화 (pulse 폭 = 1클럭 보장)
clear_pulse     <= 1'b0;
char_wr_pulse   <= 1'b0;
canvas_wr_pulse <= 1'b0;

if (write_fire) begin
    case (wr_addr_word)
        ADDR_WR_STRB: if (S_AXI_WDATA[0]) char_wr_pulse <= 1'b1;
        ...
    endcase
end

5. tb_vga.v — vga_ctrl 직접 테스트벤치

AXI 없이 vga_ctrl 모듈을 직접 구동하여 내부 버퍼 동작과 VGA 타이밍을 검증합니다. 5가지 핵심 시나리오를 순서대로 실행합니다.

테스트 시나리오

시나리오 1 — 문자 버퍼 쓰기 검증

write_char(11'd0, 8'h41);  // 주소 0에 'A' 쓰기
if (uut.char_buf[0] !== 8'h41) $fatal; // RAM 값 확인

write_char task: addr/data 설정 → char_wr_en=1 → 다음 클럭 → char_wr_en=0

시나리오 2 — 캔버스 버퍼 쓰기 검증

write_canvas(10'd0, 1'b1);  // 픽셀 0번을 흰색(1)으로 설정
if (uut.canvas_buf[0] !== 1'b1) $fatal;

시나리오 3 — clear 기능 검증

pulse_clear();             // 1클럭 clear 펄스
wait (clear_busy == 1'b1);  // clear 시작 확인
wait (clear_busy == 1'b0);  // 1200클럭 후 완료 대기
// char_buf[0]과 [1199] 모두 0x20(space)인지 확인

시나리오 4 — 라인 롤오버 검증

wait_pixel(10'd799, 10'd0);  // h=799, v=0 픽셀까지 대기
@(posedge clk);
// 다음 클럭에 h_cnt=0, v_cnt=1이어야 함
if (uut.h_cnt !== 10'd0 || uut.v_cnt !== 10'd1) $fatal;

시나리오 5 — Hsync 폭 검증

wait_pixel(10'd0, 10'd2);
// v_cnt=2 줄 전체에서 Hsync=0인 픽셀 카운트
while (uut.v_cnt == 10'd2) begin
    if (uut.pix_phase == 2'd3 && Hsync == 1'b0)
        hsync_low_pixels = hsync_low_pixels + 1;
    @(posedge clk);
end
if (hsync_low_pixels != 96) $fatal;  // 반드시 96이어야 함

6. tb_vga_axi.v — AXI 버스 통합 테스트벤치

vga_axi 전체를 테스트합니다. AXI 핸드셰이크를 직접 구동하여 레지스터 맵 동작과 VGA 타이밍을 동시에 검증합니다.

AXI 태스크 정의

task axi_write;
    // 1. awaddr + wdata 설정
    // 2. awvalid=1, wvalid=1 Assert
    // 3. awready && wready 대기 → Deassert
    // 4. bvalid 대기 → bresp == 2'b00(OKAY) 확인
endtask

task axi_read;
    // 1. araddr 설정, arvalid=1
    // 2. arready 대기 → arvalid=0
    // 3. rvalid 대기 → rdata 캡처
    // 4. rresp == 2'b00(OKAY) 확인
endtask

문자 쓰기 시퀀스

task vga_write_char;
    axi_write(6'h04, {21'd0, addr});  // CHAR_ADDR 설정
    axi_write(6'h08, {24'd0, ch});   // CHAR_DATA 설정
    axi_write(6'h0C, 32'h1);          // WR_STRB 트리거 (char_wr_en 펄스 생성)
endtask

테스트 시나리오 요약

순서동작검증 항목
1리셋 후 CTRL 읽기초기값 enable=1 확인 (rd[0]==1)
2'H', 'I' 문자 쓰기char_buf[0]=='H', [1]=='I'
3캔버스 픽셀 쓰기canvas_buf[0]==1
4CTRL에 enable+clear 동시 쓰기clear_busy 1→0 전이, 버퍼 전체 0x20
5canvas_mode 쓰고 읽기readback rd[0]==1
6라인 롤오버h=799→0, v=0→1
7Hsync 폭 계수96 픽셀 Low 확인

7. 예상 파형 설명

파형 1 — 문자 버퍼 쓰기 (write_char)

clk__‾‾__‾‾__‾‾__‾‾__‾‾__‾‾
char_wr_addr────[ 0x000 ]──────────────
char_wr_data────[ 0x41='A']──────────────
char_wr_en─────────‾‾______────────────
char_buf[0]──────────────[0x41]─────────

char_wr_en이 High인 클럭의 상승 엣지에서 char_buf에 값이 기록됩니다. 다음 클럭부터 char_buf[0]이 0x41('A')로 유지됩니다.

파형 2 — clear 동작

clk__‾‾__‾‾__‾‾...(1200클럭)...‾‾
clear──‾‾__──────────────────────
clear_busy────‾‾‾‾‾‾...(1200클럭)...____
clear_idx────[0][1][2]...[1199]────────
char_buf[n]──────────────────[0x20 ...]──

clear 펄스 직후 clear_busy=1이 되고, clear_idx가 0→1199로 증가하면서 매 클럭 char_buf에 0x20(공백)을 씁니다. 1200클럭 후 clear_busy=0으로 복귀합니다.

파형 3 — 수평 카운터 롤오버

pix_phase[0][1][2][3][0][1][2][3]...
h_cnt(phase3)[ 797 ][ 798 ][ 799 ][ 0 ][ 1 ]...
v_cnt(phase3)[ 0 ][ 0 ][ 0 ][ 1 ][ 1 ]...

h_cnt는 Phase 3에서만 증가합니다. 799 다음에 0으로 리셋되고, 동시에 v_cnt가 1 증가합니다.

파형 4 — Hsync / Vsync 생성

h_cnt[639][640..655][656..751][752..799][0]...
Hsync──────────────‾_________‾‾‾‾‾‾──────
v_cnt[479][480..489][490..491][492..524][0]...
Vsync──────────────‾_________‾‾‾‾‾‾──────

Hsync는 h_cnt가 656~751 구간에서 Low(96 픽셀). Vsync는 v_cnt가 490~491 구간에서 Low(2 라인). 모두 active-low 입니다.

파형 5 — 폰트 렌더링 파이프라인

pix_phase[ 0 ][ 1 ][ 2 ][ 3 ]
char_rd_addr[ row*40+col 계산 ]──────────
char_code_q──[RAM→0x41'A']──────────────
font_addr───────[{0x41, row}]──────────
font_data──────────────[ROM→0xFF00...]─
text_pixel_on─────────────────[ 1 or 0 ]─
vgaRGB──────────────────[fg or bg]─

Phase 0에서 char_buf를 읽어 char_code_q에 저장, Phase 1에서 font_rom에 주소 전달, Phase 2에서 font_rom 출력 안정화(1클럭 레이턴시), Phase 3에서 최종 색상 결정 및 VGA 출력.

파형 6 — AXI 쓰기 핸드셰이크 (tb_vga_axi)

S_AXI_ACLK__‾‾__‾‾__‾‾__‾‾__‾‾__‾‾
AWVALID────‾‾‾‾‾‾______────────
AWREADY──────────‾‾______────────
WVALID────‾‾‾‾‾‾______────────
WREADY──────────‾‾______────────
BVALID──────────────‾‾____────
BREADY‾‾‾‾‾‾‾‾‾‾‾‾‾‾‾‾‾‾‾‾‾‾
BRESP──────────────[00:OKAY]────

마스터가 AWVALID+WVALID를 동시에 Assert → 슬레이브(vga_axi)가 AWREADY+WREADY로 수락 → BVALID로 응답(OKAY) → BREADY로 응답 수신 완료.

시뮬레이션 팁: tb_vga.v는 $dumpvars로 VCD 파일을 생성합니다. GTKWave 등에서 h_cnt, v_cnt, Hsync, Vsync, vgaRed를 동시에 확인하면 VGA 타이밍을 시각적으로 검증할 수 있습니다.
